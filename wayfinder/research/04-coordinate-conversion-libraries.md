# Research: Coordinate conversion libraries for GPSView

Ticket: `wayfinder/tickets/04-research-coordinate-conversion-libraries.md`
Researched: 2026-07-11
Context: personal, sideload-only Android app, Kotlin + Jetpack Compose, min SDK 34, Google Play services allowed. GPSView must convert WGS84 lat/lon into UTMREF/MGRS (as used by Bavarian BOS) and Plus Codes entirely offline, and the standing decision is to drop what3words unless a genuine free offline path exists.

> Methodology note: all claims below are backed by a primary source (Maven Central artifact metadata fetched via `search.maven.org`'s API and direct HTTP HEAD/GET against `repo1.maven.org`, GitHub repository metadata fetched via `gh api`, an official Bavarian state fire-service training PDF fetched and read directly, or a vendor's own docs/ToS page) with an inline citation next to the claim. Where something could not be confirmed from a primary source, or requires on-device verification, it is flagged explicitly rather than guessed. One library's fitness (GeographicLib-Java) was checked by downloading and inspecting its actual published sources JAR, not by trusting a search-engine summary — see §1.2, which corrects a wrong secondary-source claim.

## 1. UTMREF/MGRS conversion library

### 1.1 `mil.nga:mgrs` (+ `mil.nga.mgrs:mgrs-android`) — recommended primary candidate

Developed by the **National Geospatial-Intelligence Agency (NGA)** in collaboration with BIT Systems/CACI — i.e. the same U.S. government body that co-authors the MGRS specification itself, which is about as strong a spec-conformance pedigree as a library can have. Source: repo README, fetched directly (https://raw.githubusercontent.com/ngageoint/mgrs-java/master/README.md, 2026-07-11): *"The MGRS Library was developed at the National Geospatial-Intelligence Agency (NGA) in collaboration with BIT Systems... The government has 'unlimited rights'... released under the MIT license."*

- **Repo / license / maintenance** (via `gh api repos/ngageoint/mgrs-java`, 2026-07-11): license **MIT**, `archived: false`, `pushed_at: 2024-04-03T20:45:37Z`, 25 stars, 3 open issues. The companion Android wrapper `ngageoint/mgrs-android` (`gh api repos/ngageoint/mgrs-android`): license **MIT**, `archived: false`, `pushed_at: 2024-04-04T20:26:48Z`, 11 stars, 1 open issue.
- **Maintenance caveat**: both repos are healthy (not archived, low open-issue count) but their last push was **April 2024 — roughly 27 months before this research (2026-07-11)**. This is not a red flag on its own (MGRS math is a stable, finished spec — there's little reason for churn), but it means neither library has been exercised against very recent Android/Gradle toolchain changes; do a build smoke-test before committing.
- **Maven Central coordinates and versions**, confirmed via `search.maven.org`'s Solr API (2026-07-11):
  - `mil.nga:mgrs:2.1.3` (pure Java, `packaging=jar`), published 2024-04-03.
  - `mil.nga.mgrs:mgrs-android:2.2.3` (`packaging=aar`), published 2024-04-04.
- **Size and dependency footprint**, confirmed via direct HTTP HEAD against `repo1.maven.org` and the artifacts' own POMs (2026-07-11):
  - `mgrs-2.1.3.jar`: **48.5 KB** (`content-length: 49628`). Its only runtime dependency is `mil.nga:grid:1.1.2` (also NGA/BIT Systems, also MIT), **31.3 KB** (`content-length: 32034`) — a shared grid-math library used across NGA's family of reference-system libraries (MGRS, GARS, etc.). Total footprint for pure-Java MGRS conversion: **~80 KB, zero third-party runtime dependencies.**
  - **Important dependency finding for GPSView specifically**: `mgrs-android-2.2.3.pom` declares a runtime dependency on **`com.google.android.gms:play-services-maps:18.2.0`** (fetched directly from `https://repo1.maven.org/maven2/mil/nga/mgrs/mgrs-android/2.2.3/mgrs-android-2.2.3.pom`, 2026-07-11). This is because `mgrs-android` adds **Google Maps tile-overlay rendering** of MGRS grid lines on top of the base conversion logic (its README explicitly frames this as "Draw Tile Template... See mgrs-android for a concrete example", https://raw.githubusercontent.com/ngageoint/mgrs-java/master/README.md). **GPSView uses MapLibre (per ticket 01), not Google Maps**, so pulling in `mgrs-android` would drag in the entire Play Services Maps SDK purely for a rendering feature GPSView won't use. **Recommendation: depend on the plain `mil.nga:mgrs` JVM artifact directly (it runs fine on Android, being pure Java 8+), not `mgrs-android`.**
- **API shape**, confirmed from the README's own usage examples (https://raw.githubusercontent.com/ngageoint/mgrs-java/master/README.md): `MGRS.from(Point)`, `MGRS.parse(String)`, `mgrs.toUTM()`, `utm.toMGRS()`, and precision-scoped output via `mgrs.coordinate(GridType.GZD | HUNDRED_KILOMETER | TEN_KILOMETER | KILOMETER | HUNDRED_METER | TEN_METER | METER)`. This maps directly onto the even-digit-count precision levels Bavarian BOS use (§2 below: 4/6/8/10-digit).
- **Zone 32U/33U correctness**: since NGA co-owns the MGRS spec and this library implements the standard, non-exceptional grid-zone-designation algorithm (100 km square lettering, band letters, zone numbers), there is no special-cased "Bavaria exception" to get wrong — Bavaria's grid-zone boundaries (see §2) fall on ordinary zone/band lines, not on one of the MGRS spec's known irregular zones (the Norway 31V/32V widening and the Svalbard 31X–37X band splits, both far north at 56°N–84°N, nowhere near Bavaria's 47.3°N–50.6°N). **Not independently re-verified against a live test vector at the exact Bavarian 12°E/48°N crossing point in this session — flagged for on-device/unit-test verification before shipping** (e.g. assert that a point just west of 12°E/48°N in Bavaria resolves to `32U`/`32T` and a point just east resolves to `33U`/`33T`).
- **Output formatting caveat**: the README's `MGRS.parse()` example accepts a spaced input (`"33X VG 74596 43594"`), but neither the README nor the fetched Javadoc (http://ngageoint.github.io/mgrs-java/docs/api/mil/nga/mgrs/MGRS.html) states definitively whether `toString()`/`coordinate(GridType)` **emit** spaces in the NGA/compact style or in the grouped Bavarian-BOS style (`32U NA 648 215`, see §2). **This must be verified on-device or in source** — GPSView will likely need its own thin formatting layer to reproduce the exact BOS spacing convention (zone-field, space, 100km-square, space, easting digits, space, northing digits) regardless of what the library's raw `toString()` does.

### 1.2 GeographicLib-Java — not usable for MGRS/UTM, contradicts an initial (wrong) secondary-source summary

An initial web search claimed GeographicLib-Java supports "conversions between geographic, UTM, UPS, MGRS, geocentric, and local cartesian coordinates" with a documented `MGRS` class. **This turned out to be false for the actual published Java artifact** — it describes GeographicLib's C++ core (which does have UTMUPS/MGRS/GeoCoords classes) or possibly other language bindings, not the Java port on Maven Central. Verified directly by downloading and inspecting the real sources JAR:

- Maven coordinates confirmed via Solr search API: `net.sf.geographiclib:GeographicLib-Java:2.0`, published ~2022-04-25 (`timestamp: 1650875007000`). License **MIT** — confirmed both via the official GeographicLib license page (https://geographiclib.sourceforge.io/C++/doc/LICENSE.txt: *"The MIT License (MIT). Copyright (c) 2008–2025, Charles Karney"*) and via `gh api repos/geographiclib/geographiclib` (`license: MIT`, `archived: false`, `pushed_at: 2026-06-22` — the umbrella repo is actively maintained).
- **Downloaded `GeographicLib-Java-2.0-sources.jar` directly from Maven Central and listed its contents** (`https://repo1.maven.org/maven2/net/sf/geographiclib/GeographicLib-Java/2.0/GeographicLib-Java-2.0-sources.jar`, 2026-07-11). The complete class list is: `Accumulator, Constants, GeoMath, Geodesic, GeodesicData, GeodesicLine, GeodesicMask, GeographicErr, Gnomonic, GnomonicData, Pair, PolygonArea, PolygonResult`. **There is no `MGRS`, `UTMUPS`, or `GeoCoords` class anywhere in the published Java artifact.** It implements geodesic (great-circle/ellipsoid distance and bearing) math and the Gnomonic projection only.
- The current `geographiclib/geographiclib` GitHub repo's root tree (`gh api repos/geographiclib/geographiclib/git/trees/main`) has **no `java/` directory at all**, confirming the Java port's source of truth lives elsewhere (historically SourceForge, matching the `net.sf.geographiclib` group ID) and has diverged/lagged from whatever added MGRS support in other language bindings.
- **Verdict: GeographicLib-Java is not a candidate for UTMREF/MGRS conversion**, despite what a naive search-engine summary suggests. Do not use it for this purpose in GPSView (it could still be useful later for GPS bearing/distance calculations, which is unrelated to this ticket).

### 1.3 Proj4J (`org.locationtech.proj4j`) — general CRS reprojection, no MGRS support

- Repo: https://github.com/locationtech/proj4j. `gh api repos/locationtech/proj4j`: license shows as `NOASSERTION` in GitHub's auto-detected SPDX field, but the repo's actual `LICENSE` file (fetched directly, https://raw.githubusercontent.com/locationtech/proj4j/master/LICENSE) states plainly: *"This software is licensed under the Apache 2 license... Copyright 2011-2016 Azavea"*, plus a bundled MIT-licensed portion of `proj.4` C code it ports. So: **Apache-2.0** (with an embedded MIT-licensed component), not "unknown."
- **Actively maintained**: `pushed_at: 2026-06-02T15:56:39Z` (about 5 weeks before this research), 228 stars, 29 open issues — the most recently active of the three candidates.
- Maven coordinates confirmed via Solr search: `org.locationtech.proj4j:proj4j:1.4.0` (packaging `bundle`), published 2026 (`timestamp: 1743431896529`, i.e. ~April 2025 despite the far-future-looking epoch value — treat the exact date as approximate but "recent"). Companion modules `proj4j-epsg` (EPSG code database) and `proj4j-geoapi` (GeoAPI wrapper) also exist.
- **Footprint**: `proj4j-1.4.0.jar` is **254.5 KB** (`content-length: 260641`) — several times larger than `mil.nga:mgrs`, because it's a general-purpose coordinate-reference-system reprojection engine (arbitrary proj4-string-defined CRSes), not a narrow MGRS-only tool.
- **MGRS support**: none. Searched the repo's source tree via `gh api .../git/trees/master?recursive=true` for anything MGRS-related — zero hits. Proj4J can compute a **UTM** projection (given zone/hemisphere) but has no grid-zone-designator/100km-square-letter/MGRS-string logic; GPSView would have to hand-implement the MGRS lettering scheme on top of Proj4J's raw UTM easting/northing, which is exactly the error-prone part NGA's library already solves and validates.
- **Verdict: not needed for this ticket.** Proj4J is the right tool if GPSView ever needs arbitrary CRS reprojection (e.g. Gauß-Krüger legacy coordinates, though the Bavarian BOS document below confirms Gauß-Krüger is deprecated as of 2020), but it is the wrong, heavier tool for MGRS specifically.

### 1.4 Recommendation for §1

**Use `mil.nga:mgrs:2.1.3` (plain JVM artifact, not `mgrs-android`) plus its one dependency `mil.nga:grid:1.1.2`.** Pure Java, MIT-licensed, ~80 KB combined, zero third-party runtime dependencies, authored by the U.S. government agency that co-owns the MGRS specification. Do not pull in `mgrs-android` — its only addition over the plain JVM artifact is Google Maps tile-overlay rendering, which requires `com.google.android.gms:play-services-maps` and is irrelevant to a MapLibre-based app. Before shipping: (a) unit-test the library against a known coordinate straddling Bavaria's 12°E/48°N grid-zone crossing (see §2) to confirm `32U`/`32T`/`33U`/`33T` boundaries resolve correctly; (b) confirm on-device what `MGRS.toString()`/`coordinate(GridType)` actually emit (spaced or compact) and build GPSView's own formatting layer to match the Bavarian BOS convention in §2 if needed.

## 2. Bavarian BOS UTMREF format and notation — resolved from an official primary source

This was resolved with an authoritative, directly-fetched primary source: **"Kartenkunde – Geodaten für die Feuerwehren" (Merkblatt 9.008), published by the Staatliche Feuerwehrschule Würzburg, with contribution ("Mitwirkung") from the Bayerisches Staatsministerium des Innern für Sport und Integration, the Staatliche Feuerwehrschulen Bayerns, and the Landesfeuerwehrverband Bayern e. V.** — i.e. an official state training document co-authored with the Bavarian Interior Ministry, not a secondary summary. Fetched directly (2026-07-11): https://www.lfv-bayern.de/media/filer_public/4e/03/4e03b9ae-79da-40d1-86bb-8b22943c4d6c/9008_merkblatt_kartenkunde.pdf (Version 5.0, 10/2022).

### 2.1 UTMREF = MGRS, by the document's own explicit statement

Section 6.1.1.1, verbatim (p. 21): *"Die Begriffe UTMREF (Universale Transversale Mercator Referenzsystem) und MGRS (Military Grid Reference System) sind gleichbedeutend und stellen das Koordinatenformat dar, das die Feuerwehren und alle Beteiligten im Katastrophenschutz bei der Verwendung von gedruckten topographischen Karten nutzen."* ("The terms UTMREF and MGRS are synonymous and represent the coordinate format that fire brigades and all parties in disaster response use with printed topographic maps.") This directly confirms the library chosen in §1 (an MGRS library) is the correct tool for producing Bavarian BOS "UTMREF" output — there is no separate BOS-specific grid math, only a specific notational convention layered on top of standard MGRS (below).

### 2.2 Exact digit count and spacing convention

Section 5, p. 10–11, gives the worked example of the EU's geographic midpoint near Gadheim, Bavaria: **`32U NA 648 215`**. The document breaks this down explicitly into three space-separated groups: **Zonenfeld** (`32U`) + **100-km-Quadrat** (`NA`) + **Ostwert & Nordwert** (`648` `215`), i.e. the space-separated groups are *zone-field*, *100 km square letter-pair*, *easting digits*, *northing digits* — easting and northing are each their own space-delimited group, not concatenated.

The precision/digit-count table (p. 11, verbatim):

| UTMREF example | Stellenanzahl (digit count) | Genauigkeit (precision) |
|---|---|---|
| `32U NA 64 21` | 4-stellig | 1 km |
| `32U NA 648 215` | 6-stellig | 100 m |
| `32U NA 6484 2157` | 8-stellig | 10 m |
| `32U NA 64846 21576` | 10-stellig | 1 m |

Explicit rule, verbatim (p. 11): *"Die Anzahl der Stellen einer UTMREF-Koordinate ist immer gerade, nie ungerade. Es gibt folglich keine 3-, 5-, 7- oder 9-stellige UTMREF-Koordinate!"* (digit count is always even — no 3/5/7/9-digit UTMREF coordinate ever exists; easting and northing always carry equal digit counts).

**Operational default**: the document states 6-digit (100 m) precision is the standard for fire-service operations — *"Im Rahmen von Feuerwehreinsätzen, bei denen Kartenmaterial eingesetzt wird, ist im Regelfall eine 6-stellige UTMREF-Koordinate und somit eine 100 m-Genauigkeit ausreichend"* (p. 11) — but 10-digit/1 m precision is defined and available when a GPS-derived fix (rather than a hand-read map grid) is the source, which is exactly GPSView's use case. **GPSView should default to showing/copying 10-digit (1 m) precision since it has a live GPS fix, not a hand-plotted map position, but should make the 6-digit "voice-friendly" form easily accessible too** (see next point).

**When zone-field/100km-square can be dropped**: p. 21, verbatim: *"Wenn beim Übermitteln von UTMREF-Koordinaten klar ist, in welchem Zonenfeld sowie 100-km-Quadrat man sich befindet... kann auf die Angabe von Zonenfeld... und 100-km-Quadrat... verzichtet werden."* — i.e. in radio-voice-procedure use, only the easting/northing pair is spoken once context is established, and the doc gives the exact spoken form for the example: *"vierundsechzig acht, einundzwanzig fünf"* (p. 22). This is a voice-procedure convention, not something GPSView's on-screen/copy format needs to replicate, but it's worth knowing GPSView's full-precision UTMREF string is the "written/first-transmission" form, and a truncated form is normal in practice.

### 2.3 Correction to the ticket's premise: Bavaria touches **four** grid zones, not two

The ticket frames this as "zones 32U and 33U." The primary source's own diagram (Abb. 7/Abb. 9, p. 14/16) shows this is incomplete: *"Betrachtet man nun die Abbildung 7, so ist zu erkennen, dass zwar der größte Teil der Bundesrepublik im Zonenfeld 32U liegt, Teile von Bayern aber in den Zonenfeldern 32T, 33T und 33U liegen."* (p. 14) — **Bavaria's territory actually spans all four grid-zone quadrants around the 12°E / 48°N crossing point: 32T, 32U, 33T, and 33U** (32U covering most of the state; 33U the eastern strip; 32T/33T the southernmost Alpine fringe below the 48°N band line). The crossing point itself falls inside Bavaria, roughly in the Regensburg/Cham area per the diagram. **GPSView's UTMREF conversion and any tests for zone-boundary correctness should cover all four zone fields, not just 32U/33U.**

### 2.4 Related conventions confirmed by the same document (useful context for the spec)

- **Reference datum**: ETRS89, equivalent to WGS84 for this purpose, is the current official system since 2019, having replaced the older Gauß-Krüger (GK) system; the doc explicitly says GK coordinates should no longer be used (p. 9, p. 21) — confirms GPSView needn't support GK at all.
- **Coordinate-order mnemonic**: *"Ran an den Baum, Rauf auf den Baum"* — easting (horizontal) always stated/entered before northing (vertical) (p. 10, p. 18) — same east-then-north convention as standard UTM/MGRS, no surprise, but worth matching in UI field order if GPSView ever offers manual UTMREF entry.
- **what3words is explicitly still taught to Bavarian BOS** as of this document's 2022 edition (§6.1.3, p. 24: *"Eine weitere Möglichkeit zur Ortsangabe besteht über das System what3words... Teilweise findet die Dreiwortadresse bei der Bezeichnung von Lotsenpunkten Verwendung"*), alongside a link to a "what3words für Rettungsdienst, Feuerwehr und Polizei" training video. This doesn't change GPSView's decision (see §3 — the offline/free-tier facts stand on their own), but it's worth knowing operationally that Bavarian responders may still expect a what3words string to be *readable*/shareable even if GPSView itself ends up standardizing on Plus Codes for its own display.
- **A directly relevant precedent GPSView should be aware of**: the same Bavarian Staatliche Feuerwehrschulen publish and distribute their own **"Offline Koordinaten Umrechner" (OKU-App)** — a free, browser-based, explicitly offline-capable coordinate converter between UTM, UTMREF/MGRS, and GPS decimal/DMS formats (§6.2, p. 25: *"Die App funktioniert browserbasiert... zeichnet sie sich durch ihre Offline-Funktionalität aus, d.h. das Umrechnen von Koordinaten ist auch ohne bestehende Internetverbindung möglich."*, hosted at `oku.sfs-bayern.de`). This is strong independent confirmation that Bavarian BOS themselves expect/require offline coordinate conversion tooling — exactly GPSView's own design constraint — and its on-screen field layout (Z/E/N triplets for UTM and UTMREF/MGRS side by side, screenshotted in the doc's Abb. 16/17) is a reasonable reference UI to mirror for GPSView's own coordinate display, though it is a different app and not something GPSView needs to integrate with.

## 3. Plus Codes / Open Location Code — `com.google.openlocationcode`

Confirmed directly against Google's own repo and the published Maven Central artifact.

- **Repo**: https://github.com/google/open-location-code. `gh api repos/google/open-location-code` (2026-07-11): license **Apache-2.0** (confirmed independently by fetching the raw `LICENSE` file, https://raw.githubusercontent.com/google/open-location-code/main/LICENSE: *"Apache License, Version 2.0"* verbatim). `archived: false`, `pushed_at: 2026-03-30T16:23:12Z` (about 3.5 months before this research — actively maintained), 4,332 stars, 56 open issues.
- **Maven Central coordinates**: `com.google.openlocationcode:openlocationcode`, latest version **1.0.4**, published 2019-05-27 per Solr search timestamp (`1558987246000`). The version number looks stale, but Plus Codes' encoding algorithm is a small, finished, mathematically-defined spec (grid-reference encoding of lat/lon into a base-20 alphabet) with little reason to change — the repo's continued 2026 commit activity (docs/tooling/other-language ports) without a new Java release version supports "stable, not abandoned" rather than "neglected."
- **Size and dependencies — confirmed directly from the published POM and JAR** (fetched from `repo1.maven.org`, 2026-07-11):
  - `openlocationcode-1.0.4.pom`: the only declared dependency is `junit:junit:4.12` at **`<scope>test</scope>`** — i.e. **zero runtime dependencies**.
  - `openlocationcode-1.0.4.jar`: **8.6 KB** (`content-length: 8782`). This is about as lightweight as a library can be — smaller than a single typical image asset.
  - Targets Java 1.8 source/target level per the POM's `maven-compiler-plugin` config — trivially compatible with min SDK 34.
- **Fitness for Android**: pure Java, no Android-specific dependency needed at all (it's plain algorithmic code — encode/decode lat-lon to/from a Plus Code string, no I/O, no platform APIs). Fully offline by construction: there is nothing in the API surface that performs network I/O.
- **Caveats**: the library's own GitHub issue history includes long-standing threads about publishing directly to Google's own Maven repo (`maven.google.com`) rather than only Maven Central/Sonatype (https://github.com/google/open-location-code/issues/452, https://github.com/google/open-location-code/issues/152) — not a blocker, since Maven Central is already the default repository for any standard Android Gradle setup, just worth knowing it isn't a `com.google.android.*` first-party Android artifact with the guarantees that might imply; it's a general JVM library incidentally published by Google.

**Verdict: `com.google.openlocationcode:openlocationcode:1.0.4` is an excellent fit** — Apache-2.0, 8.6 KB, zero runtime dependencies, offline-by-construction, actively maintained upstream even if the Java package version itself hasn't needed a bump since 2019.

## 4. what3words offline feasibility — standing decision (drop w3w, use Plus Codes) is confirmed correct

Checked directly against what3words' own pricing page, API licence agreement, and Enterprise Suite product pages — not secondary summaries.

### 4.1 Free tier explicitly excludes conversion

Fetched directly from https://accounts.what3words.com/select-plan (2026-07-11): the **Free** plan ($0/month) includes only **AutoSuggest** (address-guessing/autocomplete, rate-limited to 5 requests/second) — the page explicitly lists **"Convert to what3words address"** and **"Convert to coordinates"** as features that only appear starting at the paid **Basic** tier ($9.99/month for 1,000 conversions). This directly matches (and is corroborated by) the API Licence Agreement's own wording (https://what3words.com/api-licence-agreement, fetched 2026-07-11): *"Any person is entitled to sign up to our free usage plan for the what3words Main API, which does not allow for any convert-to-coordinates or convert to 3 Word Address requests."* **There is no free tier that performs the one operation GPSView needs (lat/lon ↔ 3-word-address conversion).**

### 4.2 Offline capability exists only as a paid Enterprise product, not available to individuals

Fetched directly from what3words' own Enterprise Suite pages (2026-07-11):
- https://developer.what3words.com/enterprise-suite/mobile-offline-sdk: *"The iOS and Android SDKs contains all the functionality of the what3words API, while also offering offline functionality and reduced latency."* — confirming an offline SDK does exist, but only inside the Enterprise Suite.
- https://what3words.com/products/enterprise-suite: *"The API Server and SDKs are charged via annual licence, or on a per-device basis, for example, car, phone or UAV."* No self-serve signup, no published price — the page's only call to action is *"Contact us to find out more."* No mention anywhere of a free or personal-developer tier for this product.

### 4.3 The API Licence Agreement forbids working around this by caching results

Fetched directly from https://what3words.com/api-licence-agreement (2026-07-11): *"If you need the coordinates for a 3 Word Address, you shall always make a call to our API."* Storage of previously-converted results is allowed only narrowly — *"You may only store 3 Word Addresses with their corresponding coordinates if: (a) it is for the sole purpose of improving the performance of Your Product; (b) this storage is temporary (up to a maximum of 30 calendar days)"* — and explicitly **not** as a substitute for calling the API: *"You may not store the coordinates derived from a 3 Word Address... if this circumvents the need for you or a third party to make a call to our API."* This closes off the obvious workaround of "call the API once per unique location and cache forever."

### 4.4 Emergency-services free access exists but doesn't apply here

what3words does offer free/discounted access to **emergency services organizations** and NGOs/charities (per a support article surfaced during this research, https://support.what3words.com/en/articles/2588610, and referenced again on the pricing page) — but GPSView is explicitly a **personal** app for one individual's own use, not an application built and operated on behalf of a registered emergency-services organization, so this program doesn't apply. (If GPSView were ever adopted/deployed by an actual BOS organization rather than used personally, that would be a distinct licensing conversation worth revisiting.)

**Verdict: the standing decision is confirmed correct and should be locked in.** There is no legitimate free, offline, no-network-call path to what3words conversion for a personal sideloaded app. Every path — Free tier (no conversion access at all), paid Basic/Standard/Plus/Premium tiers (network-dependent, per-conversion billed), and the offline Mobile SDK/API Server (Enterprise-only, "contact us" annual/per-device licensing) — is either unavailable or requires ongoing network access and/or a paid commercial relationship incompatible with a personal, offline-capable sideload app. **Drop what3words; ship Plus Codes (`com.google.openlocationcode`, §3) as the human-shareable short-code format.**

## Recommendation

1. **UTMREF/MGRS conversion: `mil.nga:mgrs:2.1.3`** (plain JVM artifact) plus its dependency `mil.nga:grid:1.1.2`. MIT-licensed, NGA-authored (spec-authoritative), pure Java, ~80 KB combined, zero third-party runtime dependencies. **Do not** use `mil.nga.mgrs:mgrs-android` — it exists only to add Google-Maps-tile-overlay rendering and drags in `com.google.android.gms:play-services-maps`, which is dead weight for a MapLibre-based app. **Do not** consider GeographicLib-Java (its published Java artifact has no MGRS/UTMUPS support at all — verified by inspecting its actual sources JAR) or Proj4J (no MGRS support, ~5x the footprint, general-purpose CRS tool solving a different problem).

2. **UTMREF display/copy format: match the Bavarian BOS convention exactly**, confirmed from the official Staatliche Feuerwehrschulen Bayerns / Bayerisches Staatsministerium des Innern training document (§2): space-separated groups `<Zonenfeld> <100km-Quadrat> <Ostwert> <Nordwert>`, e.g. `32U NA 648 215`, always an even total digit count split evenly between easting and northing (4/6/8/10 digits → 1 km/100 m/10 m/1 m precision). **GPSView should default to full 10-digit (1 m) precision** since it has a live GPS fix (unlike a hand-plotted map reading, where 6-digit/100 m is the taught default), and the spec must not assume the library's raw `toString()` output already matches this spacing — verify on-device and add GPSView's own formatting layer if not. The spec should also explicitly note that Bavaria spans **four** MGRS grid zones (32T, 32U, 33T, 33U — not just 32U/33U as originally assumed) and that any correctness test suite must cover all four.

3. **Plus Codes: `com.google.openlocationcode:openlocationcode:1.0.4`.** Apache-2.0, Google's own reference implementation, 8.6 KB, zero runtime dependencies, offline by construction, upstream repo actively maintained. This is an easy, low-risk inclusion.

4. **what3words: confirmed dropped, no further investigation needed.** No free tier performs conversions; the only offline path (Mobile SDK / API Server) is an Enterprise product sold via annual or per-device licence with no self-serve or personal-developer path; the licence agreement additionally forbids caching results as a substitute for live API calls.
